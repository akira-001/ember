### [ERROR] kotoba-whisper-v2 / v1 が iPad 遠距離音声で 0 segments を返す（構造的）
*Arousal: 0.9 | Emotion: Frustration→Insight*

切替後、`segments=0 lang=ja prob=1.00 duration=*s` で空文字返却が連発。以下を順次試して全部効かず：

1. `chunk_length=15 condition_on_previous_text=False`（公式モデルカード推奨）
2. `compute_type="int8"`（CPU 標準）
3. `no_speech_threshold=0.95 log_prob_threshold=-2.0 compression_ratio_threshold=3.5` 緩和
4. `vad_filter=False`
5. PCM gain 正規化（peak→0.5）
6. kotoba-v1.0 で再試行

**Web 調査で根本原因判明**：Distil-Whisper（kotoba のベース）は anti-hallucination 訓練で「ノイズには空文字」を学習している（1% noise-only training）。kotoba-v2 訓練データは ReazonSpeech（クリーン TV 音声）+ WER>10 サンプル除外 → iPad 遠距離は完全に OOD。**仕様通りの動作**。

→ 結論：パラメータ調整では直らない。コード診断で 30 分以上溶かしたが、**Web 1 検索 5 分で解決**できた。

### [INSIGHT] GitHub OpenAgents の共有ブラウザは Browserbase + Playwright
*Arousal: 0.5 | Emotion: Insight*

`openagents-org/openagents` のコード解析で判明：
- 共有ブラウザはスクラッチではなく **Browserbase（クラウド SaaS）+ Playwright（ローカル）のハイブリッド**
- Cloud モード：Browserbase の `liveUrl` を iframe で表示、Playwright が CDP 経由で操作、persistent context で認証永続
- Local モード：Playwright Chromium headless、スクショ画像で代替（ライブ感弱）
- メッセージ履歴は 3 層ストレージ（RAM 1000 件 / Hot 7 日 / Archive 180 日 gzip）
- エージェント側は MCP `workspace_get_history` 最大 500 件 + Claude Code `--resume` で session 継続

参考価値：Akiraさんの Mei + Eve + cogmem 構成は**意味記憶の質で OpenAgents を上回る**が、認証必須サイトのライブ操作は Browserbase 相当の仕組みが必要。

### [PATTERN] graceful close は 2 段階（browser.close → pgrep kill）
*Arousal: 0.9 | Emotion: Lesson*

最重要発見。`pkill -f "remote-debugging-port=9222"` で殺すと **Google 認証 Cookie（SID/HSID/__Secure-1PSID 等）がメモリから flush されずに消滅**。

**正しい手順**：
1. Playwright で `browser.close()` → CDP 切断と Cookie flush 発動
2. `pgrep -f ... | while read PID; do kill -TERM $PID; done` で残プロセス kill（既に flush 済みなので安全）

注意：
- `browser.close()` だけでは Chromium プロセスは死なない（CDP 切断のみ）
- `pgrep` 戻り値を変数経由で `kill` に渡すと zsh で複数 PID パースエラーになる（`while read` ループ必須）
- SIGKILL（`kill -9`）は flush 機会ゼロで絶対 NG

SKILL.md にコード例 + ❌ パターン明記済み。

### [ERROR] Electron 35 + http:// origin で getUserMedia が silent stream を返す
*Arousal: 0.9 | Emotion: Frustration*

Always-On hook 起動時に MediaRecorder が 1323 byte (webm header のみ) チャンクを 5秒ごとに送り続けてサーバが `BLOCKED short` で全弾。Console には `mic stream acquired tracks=1` が出てるのに RMS が常に 0.0000。

**原因切り分け**:
- マイク許可 OK (System Settings → Microphone → Ember Chat ON)
- `echoCancellation: false / noiseSuppression: false / autoGainControl: false` でも silent
- AudioContext + AnalyserNode を `silent gain → destination` で graph アクティブ化しても変化なし
- AudioContext / AnalyserNode 経路を諦めて MediaRecorder 単独に切替えても、user gesture 前は webm header だけ

**根本原因**: 旧 `packages/ember-chat/main.js` は `loadFile()` で **`file://` プロトコル**経由でレンダラを読み込んでいた。Chromium は `file://` を trusted origin として autoplay policy 適用外にする → user gesture 前の getUserMedia でも実マイクが取れていた。

新版は dashboard を `loadURL('http://localhost:3456/...')` で http:// 経由。autoplay policy が厳格に効いて silent stream が返っていた。

### [DECISION] Electron に旧版互換の webSecurity:false + permission handler を復活
*Arousal: 0.7 | Emotion: Resolution*

`autoplay-policy=no-user-gesture-required` switch だけでは getUserMedia 制約は解けず。旧 main.js (commit 9a9718f0^) との差分を取り、3点復活:

1. `webPreferences.webSecurity: false`
2. `session.setPermissionRequestHandler` で `media / microphone / audioCapture` を auto-grant
3. `session.setPermissionCheckHandler` で同 set に true

検証 (computer-use): cold restart → Listening 自動復元 → サーバ `[always_on] audio received` で BLOCKED 消える → 画面に `[STT/small] 'こういうのはただで見るの'` が listening_debug + DiagnosticMessage で表示。

**学び**: dashboard を loadURL にしたフェーズ (P6) で Electron セキュリティ posture も同時に変えてしまい、Always-On 機能が壊れていた。新規 web 機能を Electron に移植する際は **`file://` vs `http://` origin の Chromium policy 差分** を最初に check すべき。

### [PATTERN] サーバ側設定キー名 (listeningDebug) と client 側 (debugMode) の rename ミスマッチ
*Arousal: 0.5 | Emotion: Annoyance*

新 React 版が `settings.debugMode` を WS で送ったが、voice_chat サーバは `_settings.get("listeningDebug")` で listening_debug メッセージを gate していた。サーバ側ロガーに送信ログが無く、画面に何も出ない原因が分かりにくかった。

修正: `useEmberChat.updateSetting('debugMode', value)` 内で `next.listeningDebug = value` も同時セット → 1回の WS update_settings で両方のキーをサーバに送る。

旧名前を残すレガシーサーバとの sync では client 側で rename mirror が必要、というパターン。今後同種の rename をやる際は **client/server 両側の grep ですべての参照を一気に更新** が原則。
