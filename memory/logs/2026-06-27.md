### [INVESTIGATION/FIX] Eve プロアクティブ投稿に decision-log の生 JSON が漏洩 — `{`始まりだけのガードが「json」プレフィックスをすり抜け
*Arousal: 0.85 | Emotion: 根本原因特定 → 二層防御の穴を塞ぐ*

**発端**: Eve のプロアクティブ投稿で、本来 `message` フィールドだけを Slack に出すべきところ、内部の decision-log JSON（premise / inner_thought / plan / generate_score / evaluate_score / decision）が丸ごと投稿された。Akiraさんが「このJSONがSlackに投稿されるのは正しい挙動？」と指摘。**症状の鍵**は投稿テキストが `json\n{...}` で始まっていたこと（```json フェンス痕跡 or "json" プレフィックス）。

**根本原因（2層の防御がすり抜け、`packages/slack-bot/src/proactive-state.ts`）**:
1. `parseDecisionLog`(1754) が null を返す。フェンス除去 regex `/^```json?\n?/` は**バッククォート必須**で素の "json" プレフィックスには効かず、さらに LLM が message 内の `\n` エスケープを忘れて JSON が壊れると、フォールバックの jsonMatch パースも失敗して null に落ちる。
2. `extractMessage`(1851) の P4 ガード(1865) が `trimmed.startsWith('{')` だけ判定 → `json` 始まりはすり抜け、P3 plain-text フォールバックが **response 全体を message として返し** Slack 投稿。

**検証**: 再現テストを書き、`json`+壊れ JSON → action:'send' で生 JSON が message になることを実証（4/4緑）。`{`始まりなら P4 が効く＝**ガードの狭さ**が原因と確定（`{` 以外の前置きが付くと無防備）。

**修正**:
- (a) `extractMessage` P4 を**内容ベース化** — 内部キー `/"(decision|generate_score|evaluate_score|inner_thought|premise)"\s*:/` 検出 ＋ フェンス/"json" プレフィックス検出で、parse 失敗時も投稿を抑制。
- (b) `parseDecisionLog` を**頑健化** — 先頭フェンス（言語タグ全般）と "json" プレフィックスを剥がして正規化してからパース。

**結果**: 回帰テスト `packages/slack-bot/src/__tests__/proactive-json-leak.test.ts`（7件）＋ slack-bot フル suite 497件 緑。`npm run build` → `pm2 restart claude-slack-bot`（`npm run prod` = `node dist/index.js` なので build 必須）でデプロイ、クリーン起動を確認。commit ff7b626b。

**教訓**: 「raw JSON を投稿しない」ガードは `{` 始まりだけでは不十分。LLM はフェンスや前置き（"json" など）を付けてくるので、形ではなく**内容（内部キーの有無）ベースで判定**すべき。
