### Session 2: ChatGPT auth テストの Chrome 誤起動

#### [FIX] テストが Chrome を実起動していた
- 原因: `src/__tests__/chatgpt-auth.test.ts` の login テストが `child_process.exec` を未モックで呼び出し、`/codex/device → /log-in` を実起動
- 修正1: `vi.mock('child_process', () => ({ exec: vi.fn() }))` 追加で Chrome 起動抑止
- 修正2: fetch モックを 2段 → 3段（usercode / poll=authorization_code / token 交換）。token 交換 mock 不足で `tokenRes.ok` undefined だった
- 結果: 7/7 pass、Chrome 起動なしを履歴で確認

#### [INSIGHT] exec/spawn/open を含む CLI ロジックは child_process を必ずモック

### Session 3: Ember Chat Always-On 堅牢化

#### [MILESTONE] Always-On Listener watchdog + restart
- 症状: WS 再接続/再起動後に Always-On が止まり 3日連続で co_view 無音警告
- `always-on.js`: `restart()`（destroy → 300ms → start）追加、`_startWatchdog()` 30s チェック・stale 2分・auto restart 10分
- `index.html`: `ws.onopen` で既存 alwaysOn を `restart()`、`listening_stale` CSS 追加
- 教訓: destroy→new 即時連続は WebAudio 内部リソース競合で失敗。**短い delay + watchdog** が再発防止

#### [MILESTONE] wake word 誤発火 → 話者識別ゲート
- 症状: YouTube 音声「メイ、メイ、今日のスケジュールは？」が `WAKE DETECTED` → LLM reply 到達
- `app.py` L4429 にゲート追加。`speaker_result` あり & `audio_duration >= 1.5s` で `speaker != "akira"` なら `[wake blocked] non-Akira voice`
- プロファイル未登録/短音声は通過（後方互換）。whisper-serve 再起動（PID 80226）

#### [PATTERN] ambient と wake は独立経路（計測漏れ）
- /co-view-improve が wake 誤発火を計測外（ambient 集計のみ）
- skill.md に **1-D2 Wake word 誤発火分析** 追加。`WAKE DETECTED` / `WAKE BLOCKED` / unknown 話者到達 / LLM reply 件数を計測
- 教訓: 音声入力には複数の独立経路（ambient / wake / 会話モード）。改善ループ計測を 1経路に限定すると同種誤検知を見逃す
