# セッション 3: Ember Chat Always-On 堅牢化 + wake word 誤発火防止

## セッション概要
Always-On Listener の VAD silent fail（3日間断続発生）を watchdog + restart() で根治し、YouTube 音声が wake word を trigger してしまう問題を話者識別（speaker_id）ゲートで遮断。自動改善レポートが wake 経路を計測漏れしていた問題も修正。

## [MILESTONE] Ember Chat Always-On 堅牢化

### 内容
WebSocket 再接続後 / 再起動後に Always-On Listener が止まり、3日連続で co_view 無音警告が出ていた。

### 対応
- `always-on.js`: `restart()` メソッド追加（destroy → 300ms → start）。WebAudio/MediaStream の非同期解放レース対策
- `always-on.js`: `_startWatchdog()` 追加。30s ごとに `_lastAudioSendTs` を検査し、2分超で stale、10分超で auto restart
- `index.html`: `ws.onopen` で既存 alwaysOn があれば `restart()`
- `index.html`: `listening_stale` 用 CSS（薄緑、animation none）追加。state bridge にも stale を active 扱いで含める

### 教訓
destroy→new を即時連続実行すると WebAudio 内部リソース競合で失敗する。**短い delay + watchdog** のペアが再発防止に有効。

## [MILESTONE] wake word 誤発火 → 話者識別ゲート

### 内容
YouTube 音声の「メイ、メイ、今日のスケジュールは？」が mic 経由で拾われ `WAKE DETECTED` → LLM reply まで到達した。

### 対応
- `app.py` L4429 に話者ゲート追加。`speaker_result` があり `audio_duration >= 1.5s` で `speaker != "akira"` なら `[wake blocked] non-Akira voice` で拒否
- プロファイル未登録 / 短音声は従来通り通過（後方互換維持）
- whisper-serve 再起動済み（PID 80226）

### 根本原因
wake word 検出は ambient_listener をバイパスして LLM 直行する独立経路。media_likely 分類は ambient 側にしか効かない。

## [PATTERN] ambient と wake は独立経路（計測漏れ）

### 内容
/co-view-improve 自動改善レポートが wake 誤発火を計測していなかった。

### 原因
skill.md 1-D が `grep "[ambient] source: unknown intervention=reply"` のみを集計。wake 経路は ambient を通らないため集計外。

### 対応
skill.md に **1-D2: Wake word 誤発火分析** セクション追加。`WAKE DETECTED` / `WAKE BLOCKED` / unknown 話者到達 / LLM reply 件数を計測。Slack 投稿フォーマットにも「🎙️ Wake word 誤発火分析」を追加。

### 教訓
音声入力には**複数の独立経路**（ambient / wake / 会話モード）が存在。改善ループの計測を 1 経路に限定すると同種誤検知を見逃す。

  - theme-taxonomy テスト 20件 regression（朝セッションの未完了）
  - 次回 /co-view-improve サイクル（6h 毎）で 1-D2 動作確認
