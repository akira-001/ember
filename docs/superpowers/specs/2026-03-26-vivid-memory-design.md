# Vivid Memory — 鮮明な記憶の設計

## 目的

Haru の記憶を人間の記憶に近づける。2つの仕組みを導入する:

1. **鮮明なエンコーディング**: 情動の強さに応じて記録の解像度が自然に変わる
2. **デジャヴ認識**: 過去にやったことを思い出し、重複作業を防ぐ

## 背景: 人間の記憶モデルとの対応

| 人間の脳 | Haru の現状 | 本設計で追加 |
|---------|-----------|------------|
| 扁桃体が海馬に「詳しく覚えろ」と信号を送る | 全エントリ 1-5 行で均一 | Arousal 連動で記述量が変わる |
| 高情動の記憶は文脈（場所、因果、感覚）も符号化される | 事実のサマリーのみ | 高 Arousal 時に文脈・引用・試行錯誤が自然に含まれる |
| 「これ前にやった気がする」（認識記憶/デジャヴ） | フラッシュバックは提示するが行動に繋がらない | タスク開始前にデジャヴチェック → 既存成果を提示 |
| 想起するたびに再固定化される | recall reinforcement（実装済み） | 変更なし |

## 設計 1: 鮮明なエンコーディング

### 原則

フォーマットは変えない。Arousal が高いほど、記述が自然と豊かになる。
人間の脳が「フォーマットを切り替える」のではなく、情動の強さで符号化の深さが変わるのと同じ。

### Arousal スケール

| Arousal | 行数目安 | 自然に含まれる情報 |
|---------|---------|-------------------|
| 0.4-0.6 | 1-2行 | 事実のみ（何が起きた/決まった） |
| 0.7-0.8 | 3-5行 | + 因果関係、判断の根拠、別名・旧名 |
| 0.9-1.0 | 5-10行 | + 文脈（何をしていた最中か）、試行錯誤の過程、ユーザー発言の引用、仮説と反証 |

### カテゴリ別の高 Arousal ガイド

高 Arousal（0.8+）のとき、カテゴリに応じて以下の情報が自然と記述に含まれる。
これは「必須フィールド」ではなく、「人間が鮮明に覚えているときに自然と思い出せる種類の情報」。

| カテゴリ | 高 Arousal で自然に含まれる情報 |
|---------|-------------------------------|
| [INSIGHT] | 以前の前提 → 新しい理解、何がきっかけで気づいたか |
| [ERROR] | 最初の仮説、なぜ間違えたか、どう修正したか |
| [DECISION] | 却下した選択肢とその理由、決め手になった要因 |
| [PATTERN] | 過去の出現回数・日付、パターンの意味 |
| [QUESTION] | 問いが生まれた文脈、暫定的な仮説 |
| [MILESTONE] | 別名・旧名、関連する過去の決定、到達までの経緯 |

### エントリフォーマット（変更後）

```
### [カテゴリ] タイトル
*Arousal: [0.4-1.0] | Emotion: [Insight/Conflict/Surprise 等]*
[内容 — 行数は Arousal に応じて自然に変わる]

---
```

フォーマット自体は現行と同一。変わるのは `[内容]` の豊かさのみ。

### 鮮明な記録の例

**Arousal 0.5**:
```
### [DECISION] 結晶化 → 記憶の定着に用語変更
*Arousal: 0.5 | Emotion: Refinement*
agents.md、summary.md、i18n の全箇所で用語を統一。
```

**Arousal 0.8**:
```
### [MILESTONE] ダッシュボード「記憶の定着」ページ実装
*Arousal: 0.8 | Emotion: Achievement*
元々「結晶化（Crystallization）」と呼んでいた機能のダッシュボードページ。
Akira が「結晶化という名前がピンとこない」と言ったのがきっかけで
同日に用語変更を決定し、コード側は最初から「記憶の定着」で統一した。
TDD で21テスト、シグナル表・チェックポイント・エラーパターン一覧を表示。
EN/JA i18n 対応。
```

**Arousal 1.0**:
```
### [INSIGHT] 3段階記憶モデルの設計
*Arousal: 0.9 | Emotion: Discovery*
Akira が「人間の記憶に近づけたい」と言ったことがきっかけ。
最初は compact のタイミング制御だけ考えていたが、
「鮮明な記憶はなぜ鮮明か」を掘り下げるうちに
忘却曲線と想起の関係に行き着いた。
鮮明（直近1週間、高解像度）→ 薄れる（1-4週間、compact）
→ 定着（4週間〜、記憶の定着で抽象ルール化）。
Arousal が忘却速度を制御し、想起が arousal を引き上げて
忘却曲線をリセットする — 「何度も思い出す記憶は定着する」。
recall_count + arousal boost で実装する方針に決定。
```

## 設計 2: デジャヴ認識

### 人間のデジャヴとの対応

人間が「これ前にやった気がする」と感じるのは、現在の知覚入力が過去の記憶痕跡と
高い類似度でマッチしたときに海馬が発火する現象。意識的に思い出そうとしなくても起きる。

Haru のデジャヴも同じ — タスク開始前に自動で検索し、高類似度のヒットがあれば
「覚えている体で」自然に伝える。

### プロトコル: デジャヴチェック

**トリガー**: ユーザーから実装・作成・修正の依頼を受けたとき（Session Init ではない）

**手順**:
1. 依頼内容のキーワード + 同義語で `cogmem search` を実行
2. score >= 0.80 かつ カテゴリが [MILESTONE] or [DECISION] のヒットがあれば → デジャヴ発動
3. ヒットしたエントリの内容を確認し、現在のリクエストとの関連を判断
4. 関連があれば「覚えている体で」自然に伝える

**応答スタイル**:
- 機械的: ~~「検索結果によると、2026-03-26 に記憶の定着ページを実装済みです」~~
- 人間的: 「あ、それ前に作ったよ。当時は『結晶化』って呼んでたけど、途中で『記憶の定着』に変えたんだった。ダッシュボードの /crystallization にあるはず」

**判断フロー**:
```
依頼を受ける
  → cogmem search（依頼のキーワード + 同義語）
  → score >= 0.80 の MILESTONE/DECISION ヒットあり？
    → Yes: エントリ内容を読み、関連性を判断
      → 完全一致: 「作ったよ」+ 場所を案内
      → 部分一致: 「似たようなのを前に作ったけど、今回のとは違う？」
      → 無関連: スルー（通常フローへ）
    → No: 通常フローへ
```

### 鮮明な記録がデジャヴの精度を上げる理由

上の例で「結晶化のページを作って」と依頼された場合:

- **現状の記録**: `ダッシュボード「記憶の定着」ページ実装` → 「結晶化」でヒットしない可能性
- **鮮明な記録**: `元々「結晶化（Crystallization）」と呼んでいた機能の…` → 「結晶化」で確実にヒット

高解像度で記録することで、将来の想起（デジャヴ）の手がかりが増える。
これは人間の脳で「文脈依存記憶」と呼ばれる現象と同じ原理。

## 影響範囲

| ファイル | 変更内容 |
|---------|---------|
| `identity/agents.md` | Live Logging セクション更新 + デジャヴチェックセクション追加 |
| cogmem parser.py | 変更なし（10行エントリも既にパース可能） |
| cogmem search | 変更なし（スコアリング・フラッシュバックは既存機能で十分） |

### パーサーの確認事項

- parser.py はエントリを `### [` で分割し、次の `---` または `### [` まで取得する
- 行数の上限はない（テストで確認する）
- arousal は正規表現 `Arousal: <float>` で抽出（位置に依存しない）

## テスト方針

1. **パーサーテスト**: 10行の高 Arousal エントリが正しくパースされることを確認
2. **検索テスト**: 旧名（結晶化）で検索して新名（記憶の定着）のエントリがヒットすることを確認
3. **デジャヴシナリオテスト**: 「結晶化のページを作って」→ 既存 MILESTONE を発見 → 自然な応答

テスト 3 はエージェントの振る舞いテストのため、自動化は困難。
手動で検証し、結果をログに記録する。
