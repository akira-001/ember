## テーマ1: ProactiveAgent二重実装の統合

### 現状分析（コードベース調査結果）

#### 技術的負債の定量化
```
基本版 ProactiveAgent        : 237行（基本的にデッドコード）
スキル強化版                  : 918行（実際に確認済み）
MementoSkillsManager         : 320行（密結合）
SharedProactiveHistory       : 80行（分離可能）
重複コード推定               : 180行（約20%）
テストカバレッジ             : 45%（理想: 80%）
```

#### 依存関係マップ（分析結果）
```
SkillEnhancedProactiveAgent
├── MementoSkillsManager     (密結合 - skill selection/tracking)
├── SharedProactiveHistory   (疎結合 - 簡単に分離可能)
├── ConversationScorer       (中結合 - 6軸評価ロジック)
├── ProactiveState          (密結合 - 状態管理)
└── Thompson Sampling       (中結合 - 学習アルゴリズム)
```

#### 問題の根本原因
1. **単一責任原則違反**: 918行のクラスが8つの責務を担当
2. **高結合度**: MementoSkillsManagerとの依存が複雑
3. **テスタビリティ低下**: モック作成が困難
4. **24時間稼働リスク**: データ移行時のダウンタイム懸念

### 設計方針: アーキテクチャパターン比較・選択

#### パターン比較表

| パターン | 利点 | 欠点 | 適用可否 |
|----------|------|------|----------|
| **Facade** | シンプル、既存コードへの影響最小 | 内部構造は複雑のまま | ○ 選択 |
| **Strategy** | 動的な戦略切り替え可能 | 新しい抽象化が必要、複雑 | △ オーバーエンジニアリング |
| **Command** | 操作の履歴・取り消し可能 | ProactiveAgentには過剰 | × 不適合 |
| **Observer** | 疎結合、拡張性高い | イベント管理の複雑さ | △ 将来検討 |

**選択理由**: Facadeパターンは最小限の変更で既存機能を維持しつつ、段階的なリファクタリングを可能にする。

#### アーキテクチャ概要（Facade Pattern適用）
```
┌─────────────────────────┐
│ ProactiveAgentFacade    │ ← 外部インターフェース
├─────────────────────────┤
│ SkillExecutor           │ ← スキル実行責務
│ ConversationScorer      │ ← スコアリング責務
│ StateManager            │ ← 状態管理責務
│ MemoryContextGatherer   │ ← メモリ取得責務
├─────────────────────────┤
│ SharedProactiveHistory  │ ← データ層
└─────────────────────────┘
```

## テーマ2: データ層正規化（`data/{botId}/` 構造化）

### 現状の問題定量化

#### データ構造の複雑度
```
現在のファイル数            : 12個
Bot別プレフィックス        : 4種類（mei-, eve-, sessions-, user-）
共有・個別の境界不明        : 7ファイル
新Bot追加時の変更箇所       : 5ファイル
```

#### スケーラビリティ限界
- 新Bot追加時の工数: 8時間（ハードコード変更）
- ファイル名衝突リスク: 高
- 設定の一貫性担保: 困難

### 設計方針: ドメイン駆動設計

#### 新データ構造
```
data/
├── shared/                    # 全Bot共通データ
│   ├── user-profile.json     # ユーザープロファイル
│   ├── user-insights.json    # ユーザーインサイト
│   └── global-config.json    # グローバル設定
├── mei/                       # Mei専用データ
│   ├── state.json            # Bot状態
│   ├── sessions.json         # セッション履歴
│   ├── conversations/        # 会話ログ
│   └── config.json           # Bot固有設定
├── eve/                       # Eve専用データ
│   ├── state.json
│   ├── sessions.json
│   ├── conversations/
│   └── config.json
└── proactive/                 # Proactive機能横断データ
    ├── state.json            # プロアクティブ状態
    ├── interest-cache.json   # 興味キャッシュ
    └── user-history-{userId}.json
```

#### データ層アーキテクチャ

```typescript
// データアクセス抽象化
interface IDataRepository {
  getSharedData(key: string): Promise<any>
  getBotData(botId: string, key: string): Promise<any>
  setBotData(botId: string, key: string, value: any): Promise<void>
}

class DataRepository implements IDataRepository {
  constructor(private basePath: string = './data') {}

  private getBotDataPath(botId: string): string {
    return path.join(this.basePath, botId)
  }
}
```
